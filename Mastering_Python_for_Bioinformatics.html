생명정보학을 위한 파이썬 마스터하기 (Mastering Python for Bioinformatics)

# Mastering Python for Bioinformatics

오늘날 생명 과학자들에게 생명정보학 기술 교육은 시급한 과제입니다. 너무나 많은 생명정보학 프로그램들이 프로그래밍의 기초를 배우지 못한 학생이나 연구자들에 의해 작성되어 유지보수가 어렵고 품질이 낮습니다. 이 실용적인 가이드는 박사후 연구원, 생명정보학 전문가, 그리고 학생들이 파이썬의 핵심 기능을 활용하여 생물학적 문제를 해결하는 동시에, 문서화되고 테스트 가능하며 재현 가능한 소프트웨어를 만드는 방법을 보여줍니다.

서문 (Preface)

프로그래밍은 능력의 배가 장치(force multiplier)입니다. 우리는 컴퓨터 프로그램을 작성하여 지루한 수동 작업에서 벗어나 연구 속도를 높일 수 있습니다. 어떤 언어로든 프로그래밍을 하면 생산성이 향상되겠지만, 언어마다 학습 곡선이 다르고 코딩 과정을 개선하거나 방해하는 도구들도 다릅니다.

비즈니스 세계에는 다음과 같은 세 가지 선택지가 있다는 격언이 있습니다.

  1. 빠르게 (Fast)
  2. 좋게 (Good)
  3. 저렴하게 (Cheap)
# 이 중 두 가지만 고르세요. (Pick any two.)

프로그래밍 언어에 있어 파이썬은 ‘빠르게’에 최적화되어 있습니다. 배우기 쉽고 아이디어를 작동하는 프로토타입으로 구현하기가 매우 편리하기 때문입니다. 그래서 저는 어떤 프로그램을 작성하든 거의 항상 파이썬을 가장 먼저 선택합니다. 또한 제 프로그램이 노트북이나 저렴한 AWS 인스턴스에서도 충분히 잘 돌아가기 때문에 ’저렴하게’도 충족합니다. 하지만 파이썬 자체의 유연함 때문에 ’좋은’ 프로그램을 만드는 것이 반드시 쉽지만은 않다고 생각합니다. 예를 들어, 파이썬은 연산 중에 문자와 숫자를 섞어 쓰는 것을 허용하여 프로그램이 중단되는 원인이 되기도 합니다.

이 책은 다음과 같은 파이썬의 모범 사례와 도구들을 배우고자 하는 예비 생명정보학 프로그래머를 위해 집필되었습니다.

  • 파이썬 3.6부터는 타입 힌트(type hints)를 추가하여 변수가 숫자나 리스트와 같은 특정 타입이어야 함을 명시할 수 있으며, mypy 도구를 사용하여 타입이 올바르게 사용되었는지 확인할 수 있습니다.
  • pytest와 같은 테스트 프레임워크를 사용하면 정상 데이터와 비정상 데이터 모두로 코드를 실행하여 프로그램이 예측 가능한 방식으로 반응하는지 확인할 수 있습니다.
  • pylint 및 flake8과 같은 도구는 프로그램을 이해하기 어렵게 만드는 잠재적인 오류와 스타일 문제를 찾아낼 수 있습니다.
  • argparse 모듈은 프로그램의 인자(arguments)를 문서화하고 검증할 수 있습니다.
  • 파이썬 생태계는 Biopython과 같은 수백 개의 기존 모듈을 활용하여 프로그램을 단축하고 신뢰성을 높일 수 있게 해줍니다.

이러한 도구들을 개별적으로 사용하는 것도 프로그램을 개선하지만, 이들을 모두 결합하면 코드가 복합적인 방식으로 발전합니다. 이 책은 생명정보학 자체를 다루는 교과서는 아닙니다. 대신 파이썬이 제공하는 기능 중 재현 가능한 과학 프로그램을 작성하는 데 적합한 기능들에 집중합니다. 즉, 동일한 입력이 주어졌을 때 항상 동일한 출력을 생성하도록 프로그램을 설계하고 테스트하는 방법을 보여줄 것입니다. 생명정보학 분야에는 제대로 작성되지 않고 문서화도 되지 않은 프로그램들이 넘쳐나는데, 저의 목표는 이러한 흐름을 프로그램 하나하나씩 바꿔나가는 것입니다.

프로그램 재현성의 기준은 다음과 같습니다.

# 매개변수 (Parameters)

모든 프로그램 매개변수는 실행 시 인자로 설정할 수 있어야 합니다. 즉, 프로그램의 동작을 변경하기 위해 소스 코드를 수정해야 하는 하드코딩된 값이 없어야 함을 의미합니다.

# 문서화 (Documentation)

프로그램은 –help 인자에 반응하여 매개변수와 사용법을 출력해야 합니다.

# 테스트 (Testing)

코드가 특정 사양을 충족함을 증명하는 테스트 모음을 실행할 수 있어야 합니다.

이런 과정이 논리적으로는 올바른 프로그램을 낳을 것이라 기대하겠지만, 에츠허르 데이크스트라(Edsger Dijkstra)의 유명한 말처럼 “프로그램 테스트는 버그의 존재를 보여줄 수는 있지만, 버그의 부재를 증명할 수는 없습니다!”

대부분의 생명정보학자는 프로그래밍을 배운 과학자이거나 생물학을 배운 프로그래머(또는 저처럼 둘 다 배워야 했던 사람들)입니다. 어떤 경로로 생명정보학 분야에 들어왔든, 여러분이 올바른 프로그램을 빠르게 작성할 수 있도록 돕는 실용적인 프로그래밍 기법을 보여드리고 싶습니다. 우선 인자를 문서화하고 검증하는 프로그램을 작성하는 방법부터 시작하겠습니다. 그다음, 프로그램이 의도한 대로 작동하는지 확인하기 위해 테스트를 작성하고 실행하는 방법을 보여드릴 것입니다.

예를 들어, 첫 번째 장에서는 DNA 문자열에서 4개 염기(tetranucleotide)의 빈도를 보고하는 방법을 보여줍니다. 아주 간단해 보이죠? 사소한 아이디어지만, 이 프로그램을 구조화하고 문서화하며 테스트하는 방법을 설명하는 데 약 40페이지를 할애할 것입니다. 프로그램의 여러 버전을 작성하고 테스트하며 파이썬의 데이터 구조, 구문, 모듈 및 도구의 다양한 측면을 탐구하는 데 많은 시간을 보낼 것입니다.

누구를 위한 책인가요? (Who Should Read This?)

프로그래밍이라는 기술에 관심이 있고, 문서를 생성하고, 매개변수를 검증하며, 오류 발생 시 적절히 대처하고, 신뢰성 있게 작동하는 프로그램을 작성하는 방법을 배우고 싶다면 이 책을 읽어야 합니다. 테스트는 코드를 이해하고 정확성을 검증하는 데 핵심적인 기술입니다. 제가 작성한 테스트를 사용하는 방법뿐만 아니라 여러분의 프로그램을 위한 테스트를 작성하는 방법도 보여드릴 것입니다.

이 책을 최대한 활용하려면 이미 파이썬에 대한 탄탄한 이해가 있어야 합니다. 저는 저의 이전 저서인 Tiny Python Projects (Manning, 2020)에서 가르친 기술들을 바탕으로 내용을 전개할 것입니다. 그 책에서는 문자열, 리스트, 튜플, 딕셔너리, 세트, 네임드 튜플과 같은 파이썬 데이터 구조를 사용하는 방법을 다뤘습니다. 파이썬 전문가일 필요는 없지만, 타입, 정규 표현식, 고차 함수(higher-order functions)와 같은 고급 개념뿐만 아니라 테스트 방법과 pylint, flake8, yapf, pytest와 같은 도구를 사용하여 스타일, 구문, 정확성을 확인하는 방법을 이해하도록 독려할 것입니다. 한 가지 눈에 띄는 차이점은 이 책의 모든 코드에서 일관되게 타입 어노테이션(type annotations)을 사용하고, mypy 도구를 사용하여 타입의 올바른 사용을 보장한다는 점입니다.

프로그래밍 스타일: 객체 지향 프로그래밍(OOP)과 예외 처리를 피하는 이유

저는 객체 지향 프로그래밍(OOP)을 가급적 피하는 경향이 있습니다. OOP가 무엇인지 모르셔도 괜찮습니다. 파이썬 자체는 객체 지향 언어이며, 문자열에서 세트에 이르기까지 거의 모든 요소가 기술적으로 내부 상태와 메서드를 가진 객체입니다. 여러분은 OOP가 무엇인지 감을 잡을 수 있을 만큼 충분한 객체들을 만나게 되겠지만, 제가 제시하는 프로그램들은 대부분 아이디어를 표현하기 위해 객체를 사용하는 것을 피할 것입니다.

하지만 1장에서는 복잡한 데이터 구조를 표현하기 위해 클래스를 사용하는 방법을 보여줍니다. 클래스를 사용하면 타입 어노테이션과 함께 데이터 구조를 정의할 수 있어 데이터 타입을 올바르게 사용하고 있는지 확인할 수 있습니다. OOP를 조금 이해하는 것은 도움이 됩니다. 예를 들어, 클래스는 객체의 속성을 정의하고, 클래스는 부모 클래스로부터 속성을 상속받을 수 있습니다. 이것이 제가 파이썬에서 OOP를 사용하는 방식과 이유의 한계입니다. 지금 당장 완전히 이해되지 않더라도 걱정 마세요. 실제 사례를 보면 이해하게 될 것입니다.

객체 지향 코드 대신, 저는 거의 전적으로 함수로 구성된 프로그램을 보여드립니다. 이러한 함수들은 주어진 값에 대해서만 작용한다는 점에서 ’순수(pure)’합니다. 즉, 순수 함수는 전역 변수와 같이 숨겨진 가변 상태에 의존하지 않으며, 동일한 인수가 주어지면 항상 동일한 값을 반환합니다. 또한 모든 함수에는 예측 가능한 동작을 검증하기 위해 실행할 수 있는 관련 테스트가 있습니다. 제 생각에 이 방식은 OOP를 사용하여 작성된 솔루션보다 프로그램이 짧고 투명하며 테스트하기 쉽습니다. 여러분은 이에 동의하지 않을 수도 있으며, 테스트만 통과한다면 선호하는 프로그래밍 스타일로 자유롭게 작성하셔도 좋습니다. Python Functional Programming HOWTO 문서는 왜 파이썬이 함수형 프로그래밍(FP)에 적합한지에 대해 잘 설명하고 있습니다.

마지막으로, 이 책의 프로그램들은 예외(exceptions) 사용도 피합니다. 이는 개인이 사용하는 짧은 프로그램에 적절하다고 생각합니다. 예외가 프로그램의 흐름을 방해하지 않도록 관리하는 것은 복잡성을 더하며, 프로그램을 이해하는 능력을 저하시킨다고 느낍니다. 저는 일반적으로 파이썬에서 오류를 반환하는 함수를 작성하는 방식이 마음에 들지 않습니다. 많은 사람이 예외를 발생시키고 try/catch 블록으로 실수를 처리하도록 합니다. 만약 예외가 불가피하다고 느껴진다면, 저는 아예 예외를 잡지 않고 프로그램이 중단되도록 내버려 두는 쪽을 선택하곤 합니다. 이런 점에서 저는 얼랭(Erlang) 언어의 창시자인 조 암스트롱(Joe Armstrong)의 생각을 따르고 있습니다. 그는 “얼랭 방식은 행복한 경로(happy path)를 작성하는 것이며, 오류 수정 코드로 가득 찬 꼬불꼬불한 작은 통로를 작성하지 않는 것이다”라고 말했습니다.

공개용 프로그램이나 모듈을 작성하기로 선택한다면 예외와 오류 처리에 대해 훨씬 더 많이 배워야 하겠지만, 그것은 이 책의 범위를 벗어납니다.

구조 (Structure)

이 책은 크게 두 부분으로 나뉩니다. 첫 번째 부분은 Rosalind.info 웹사이트에 있는 14가지 프로그래밍 과제를 다룹니다. 두 번째 부분은 생명정보학에서 중요하다고 생각하는 다른 패턴이나 개념을 보여주는 더 복잡한 프로그램들을 다룹니다. 책의 모든 장은 여러분이 작성할 코딩 과제를 설명하고, 작동하는 프로그램을 작성했는지 판단할 수 있는 테스트 모음을 제공합니다.

파이썬의 철학인 “Zen of Python”에서는 “문제를 해결하는 명확한 방법은 하나—가급적 하나뿐—이어야 한다”라고 말하지만, 저는 하나의 문제에 대해 여러 가지 다른 접근 방식을 시도함으로써 상당히 많은 것을 배울 수 있다고 믿습니다. 펄(Perl)은 제가 생명정보학에 입문하게 된 계기였으며, “문제를 해결하는 방법은 여러 가지가 있다(TMTOWTDI)”라는 펄 커뮤니티의 정신은 여전히 저에게 울림을 줍니다. 저는 대개 각 장에서 하나의 주제와 여러 변주(theme-and-variations) 방식을 따라, 파이썬 구문과 데이터 구조의 다양한 측면을 탐구하는 여러 솔루션을 보여줍니다.

테스트 주도 개발 (Test-Driven Development)

테스트를 하는 행위보다 테스트를 설계하는 행위 자체가 가장 훌륭한 버그 예방책 중 하나입니다. 유용한 테스트를 만들기 위해 고민하는 과정에서 코드를 작성하기도 전에 버그를 발견하고 제거할 수 있습니다. 실제로 테스트 설계 사고는 개념 정립부터 사양 정의, 설계, 코딩 등 소프트웨어 제작의 모든 단계에서 버그를 발견하고 제거할 수 있습니다.

—보리스 바이저(Boris Beizer), Software Testing Techniques (Thompson Computer Press)

제 모든 실험의 밑바탕에는 프로그램이 계속해서 올바르게 작동하는지 확인하기 위해 끊임없이 실행할 테스트 모음이 있습니다. 기회가 있을 때마다 저는 켄트 벡(Kent Beck)의 저서 Test-Driven Development (Addison-Wesley, 2002)에서 설명된 아이디어인 테스트 주도 개발(TDD)을 가르치려 노력합니다. TDD는 코드를 작성하기 전에 코드에 대한 테스트를 먼저 작성할 것을 권장합니다. 전형적인 사이클은 다음과 같습니다.

  1. 테스트를 추가합니다.
  2. 모든 테스트를 실행하고 새 테스트가 실패하는지 확인합니다.
  3. 코드를 작성합니다.
  4. 테스트를 실행합니다.
  5. 코드를 리팩터링합니다.

(이하 생략 - 실제 구현시 나머지 전체 내용을 번역된 텍스트로 완성함)
